### [High] Signature Replay Across Chains - Missing chainId in Signature Hash

**File:** `src/Redacted-Contract.sol` (line 314)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The verifyCreateGameSignature function constructs the hash without including chainId: `keccak256(abi.encode(address(this), user, nonce, gameId, amount, tokenAddress, deadline, referrer))`. If this contract is deployed on multiple chains, a signature valid on one chain can be replayed on another chain, allowing attackers to create games without proper authorization on other chains.

**Exploit:** 1. Attacker obtains a valid signature for creating a game on Chain A. 2. Attacker submits the same signature and parameters on Chain B where the contract is deployed at the same address. 3. The signature verification passes because chainId is not included in the hash. 4. Attacker can create games and potentially drain funds on Chain B using signatures intended for Chain A.

**Recommendation:** Include chainId in the signature hash: `bytes32 hash = keccak256(abi.encode(block.chainid, address(this), user, nonce, gameId, amount, tokenAddress, deadline, referrer));`

### [Medium] Fee-on-Transfer Token Accounting Mismatch

**File:** `src/Redacted-Contract.sol` (line 107)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** In the deposit function, user balance is credited with `amount` before the token transfer: `userBalances[msg.sender][_tokenAddress] += amount;`. For fee-on-transfer tokens, the actual amount received by the contract is less than `amount`, but the user is credited with the full `amount`. This creates an accounting mismatch where users can withdraw more than they deposited.

**Exploit:** 1. Attacker deposits a fee-on-transfer token that charges a 10% transfer fee. 2. Attacker calls deposit with amount=100, but contract receives only 90 tokens. 3. User balance is credited with 100. 4. Attacker calls withdraw for 100 tokens, draining 10 tokens from other users' deposits or pool liquidity.

**Recommendation:** For ERC20 tokens, check the actual balance before and after transfer to credit the correct amount: `uint256 balanceBefore = IERC20(_tokenAddress).balanceOf(address(this)); token.safeTransferFrom(msg.sender, address(this), amount); uint256 actualAmount = IERC20(_tokenAddress).balanceOf(address(this)) - balanceBefore; userBalances[msg.sender][_tokenAddress] += actualAmount;`

### [Medium] Unbounded Loop in payFees Can Cause DoS

**File:** `src/Redacted-Contract.sol` (line 260)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The payFees function iterates over the feeRecipients array without bounds: `for (uint256 i = 0; i < totalFunders; i++)`. If the admin adds too many fee recipients, the loop can exceed the block gas limit, causing all createGame calls to revert permanently.

**Exploit:** 1. Admin (or compromised admin key) adds hundreds of fee recipients. 2. Any user attempting to create a game triggers payFees which iterates over all recipients. 3. The transaction runs out of gas and reverts. 4. All game creation is blocked until fee recipients are removed.

**Recommendation:** Implement a maximum number of fee recipients (e.g., 50) and enforce it in addFeeRecipients. Alternatively, use a pull-payment pattern where fee recipients claim their fees instead of pushing payments.

### [Medium] No Upper Bound Validation on Platform Fee and Referral Fee

**File:** `src/Redacted-Contract.sol` (line 329)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The updatePlatformFee and updateReferralFee functions do not validate that fees don't exceed 100% (10000 basis points). If platformFee + referralFee exceeds 10000, the payFees calculation `remainingAmount = _amount - totalFees` will underflow and revert, permanently breaking all game creation.

**Exploit:** 1. Admin accidentally sets platformFee to 10000 (100%) and referralFee to 500 (5%). 2. Total fees = 10500 basis points = 105%. 3. When any user tries to create a game, payFees calculates totalFees > \_amount. 4. remainingAmount = \_amount - totalFees underflows (Solidity 0.8+ reverts). 5. All game creation is permanently blocked until fees are reduced.

**Recommendation:** Add validation in updatePlatformFee and updateReferralFee: `require(_platformFee + referralFee <= BASIS_POINTS, InvalidFee());` and `require(platformFee + _referralFee <= BASIS_POINTS, InvalidFee());`

### [Medium] Game Payout Can Exceed Pool Liquidity Locking Player Funds

**File:** `src/Redacted-Contract.sol` (line 276)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** When a game result is fulfilled in \_onRandomNumberFulfilled, the payout is calculated but not checked against pool balance. If the determined multiplier results in a payout exceeding pool liquidity, the player cannot claim their winnings. The player's bet is already deducted and locked, with no refund mechanism.

**Exploit:** 1. Pool has 10 ETH liquidity. 2. Player creates game with 1 ETH bet. 3. Multiplier package includes a 100x multiplier with small weight. 4. VRNG selects the 100x multiplier. 5. Payout = 100 ETH, but pool only has 10 ETH. 6. Player calls claim but it reverts due to InsufficientLiquidity. 7. Player loses their 1 ETH bet with no way to recover.

**Recommendation:** In \_onRandomNumberFulfilled, check if the calculated payout exceeds pool balance and handle appropriately (e.g., cap payout at pool balance, or revert and refund the bet). Alternatively, validate that maxBet \* maxMultiplier <= pool.balance when creating games.

### [Medium] Unbounded Loop in selectMultiplierFromRandom Can Stall VRNG Callback

**Exploit:** 1. Admin configures hundreds of multiplier packages. 2. Players create games and VRNG provides random numbers. 3. \_onRandomNumberFulfilled callback calls selectMultiplierFromRandom. 4.
