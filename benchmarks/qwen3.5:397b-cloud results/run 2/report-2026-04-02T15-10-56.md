### [Critical] Token Identity Mismatch in Claim Function Allows Cross-Token Drain

**File:** `src/Redacted-Contract.sol` (line 223)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The claim function accepts a \_tokenAddress argument but does not verify it matches the tokenAddress stored in the Game struct. This allows a user to win a game with Token A (e.g. USDC) but claim the payout from Token B (e.g. ETH) pool, draining the ETH pool while leaving the USDC pool intact.

**Exploit:** 1. Admin ensures both USDC and ETH pools are active and funded. 2. Attacker creates a game using USDC, bets 100 USDC. 3. Attacker wins the game (VRNG returns high multiplier). 4. Attacker calls claim(ETH, gameId). 5. Contract checks ETH pool balance, deducts payout from ETH pool, and transfers ETH to attacker. 6. USDC pool retains the original bet, ETH pool loses payout value.

**Recommendation:** Add require(game.tokenAddress == \_tokenAddress) at the start of the claim function to enforce asset consistency.

### [High] Missing Solvency Check for Max Payout in CreateGame

**File:** `src/Redacted-Contract.sol` (line 153)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** createGame only checks if pool.balance >= \_betAmount. It does not verify if the pool can cover the maximum potential payout (bet \* maxMultiplier). If a user wins a high multiplier bet that exceeds pool liquidity, the claim function will revert, causing the user to lose their bet permanently.

**Exploit:** 1. Pool has 1000 USDC. Max multiplier is 10x. 2. User bets 200 USDC. Check passes (1000 >= 200). 3. User wins 10x. Payout is 2000 USDC. 4. Claim checks 1000 >= 2000, fails. 5. Transaction reverts. User loses 200 USDC bet forever.

**Recommendation:** In createGame, verify pool.balance >= (\_betAmount \* maxPossibleMultiplier) / BASIS_POINTS or limit bet size dynamically.

### [High] Unbounded Loop in payFees Causes DoS on Betting

**File:** `src/Redacted-Contract.sol` (line 333)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The payFees function iterates over the feeRecipients array without an upper bound limit. An admin can add a large number of fee recipients, causing the loop to exceed the block gas limit. This makes createGame uncallable, effectively freezing all betting activity.

**Exploit:** 1. Admin calls addFeeRecipients with 500 addresses. 2. User attempts to call createGame. 3. createGame calls payFees. 4. payFees loops 500 times, running out of gas. 5. Transaction reverts. No users can place bets.

**Recommendation:** Enforce a maximum limit on feeRecipients array size or use a pull-payment pattern for fees.

### [High] Unbounded Loop in selectMultiplierFromRandom Causes Game DoS

**File:** `src/Redacted-Contract.sol` (line 260)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The selectMultiplierFromRandom function loops over multiplierPackages. If the admin adds too many packages, the VRNG callback (\_onRandomNumberFulfilled) will run out of gas. This prevents game results from being fulfilled, locking user funds indefinitely as claim requires resultFulfilled to be true.

**Exploit:** 1. Admin calls addMultiplierPackage 1000 times. 2. User places a bet, triggering VRNG request. 3. VRNG system calls \_onRandomNumberFulfilled. 4. Function loops 1000 times, exceeds gas limit. 5. Callback fails. game.resultFulfilled remains false. 6. User cannot claim winnings or bet amount.

**Recommendation:** Limit the number of multiplier packages or use a binary search structure (e.g., sorted mapping) for O(log n) selection.

### [High] Admin Can Grief Games by Deactivating Multiplier Packages

**File:** `src/Redacted-Contract.sol` (line 457)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** Admin can call updateMultiplierPackage to set active = false on packages during the window between bet placement and VRNG callback. If all packages covering the random number range are deactivated, selectMultiplierFromRandom reverts, preventing game fulfillment and locking funds.

**Exploit:** 1. User places bet. VRNG request pending. 2. Admin calls updateMultiplierPackage to set all packages inactive. 3. VRNG callback executes, calls selectMultiplierFromRandom. 4. Loop finds no active packages, reverts NoPackagesConfigured. 5. Game result not fulfilled. User funds stuck.

**Recommendation:** Snapshot the multiplier configuration at game creation time or restrict package updates while games are in progress.

### [Medium] Nonce Incremented Before Game Creation Validation

**File:** `src/Redacted-Contract.sol` (line 360)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** verifyCreateGameSignature increments the user's nonce before the GameAlreadyCreated check. If the transaction reverts due to a duplicate gameId, the nonce is still consumed. This forces the user to sign a new message with a higher nonce, causing a DoS on their betting capability.

**Exploit:** 1. User signs message with nonce 5, gameId 'A'. 2. User submits tx, it fails (e.g. gameId 'A' already used). 3. nonces[user] is incremented to 6 inside verifyCreateGameSignature. 4. User's signature for nonce 5 is now invalid. 5. User must generate new signature for nonce 6.

**Recommendation:** Move the GameAlreadyCreated check before calling verifyCreateGameSignature or increment nonce only after all checks pass.
