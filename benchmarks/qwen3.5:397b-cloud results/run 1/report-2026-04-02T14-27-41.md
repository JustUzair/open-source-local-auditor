### [High] Reentrancy in createGame() via payFees() external calls

**File:** `src/Redacted-Contract.sol` (line 202)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The createGame() function calls payFees() which performs multiple external token transfers via tokenTransfer() BEFORE updating pool.balance. An attacker can reenter during these transfers and manipulate the pool state before it's committed.

**Exploit:** 1. Attacker creates a malicious ERC20 token with a hook on transfer. 2. Call createGame() with this token. 3. During payFees() tokenTransfer to attacker-controlled fee recipient, reenter createGame() or claim(). 4. The pool.balance hasn't been updated yet (line 203 executes after payFees returns), allowing the attacker to withdraw or claim based on stale pool state.

**Recommendation:** Update pool.balance BEFORE calling payFees(). Follow CEI pattern: update all state variables before any external calls.

### [High] Missing chainId in signature verification enables cross-chain replay

**File:** `src/Redacted-Contract.sol` (line 335)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The verifyCreateGameSignature() function creates a signature hash without including chainId. Signatures created for one chain can be replayed on any other chain where this contract is deployed.

**Exploit:** 1. Attacker obtains a valid signature for createGame() on Chain A. 2. Deploy identical contract on Chain B. 3. Replay the same signature on Chain B with the same parameters. 4. The signature verifies successfully since chainId is not part of the hash, allowing unauthorized game creation.

**Recommendation:** Include chainId in the signature hash: keccak256(abi.encode(chainId, address(this), user, nonce, gameId, amount, tokenAddress, deadline, referrer))

### [High] Pool balance checks can be bypassed via direct token transfers

**File:** `src/Redacted-Contract.sol` (line 377)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** createNewPool() and removePool() check pool.balance == 0 using internal accounting. However, anyone can send tokens directly to the contract address, bypassing this check and permanently locking the ability to create or remove pools for that token.

**Exploit:** 1. Admin attempts to createNewPool() for token X. 2. Attacker sends 1 wei of token X directly to the contract address. 3. The contract's actual token balance is now > 0, but pool.balance (internal accounting) is still 0. 4. If removePool() is later called, it will succeed despite tokens being stuck. Conversely, if pool.balance tracking is somehow manipulated, createNewPool() could be blocked permanently.

**Recommendation:** Check actual token balance via IERC20(token).balanceOf(address(this)) instead of internal accounting. Or use != 0 instead of == 0 and handle edge cases.

**File:** `src/Redacted-Contract.sol` (line 260)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The selectMultiplierFromRandom() function loops over all multiplierPackages without an upper bound. An admin can add unlimited packages, causing this function to exceed block gas limit and making game resolution impossible.

**Exploit:** 1. Attacker gains ADMIN_ROLE (or is admin). 2. Call addMultiplierPackage() hundreds of times to bloat the array. 3. When \_onRandomNumberFulfilled() is called, selectMultiplierFromRandom() runs out of gas. 4. All games become unresolvable, locking user funds permanently.

### [Medium] Block timestamp used for deadline check is manipulable

**File:** `src/Redacted-Contract.sol` (line 176)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The createGame() function uses block.timestamp for deadline validation. Miners/validators can manipulate timestamps within a small window to invalidate or extend deadlines.

**Exploit:** 1. Attacker is a miner/validator or colludes with one. 2. User submits a createGame() transaction with a tight deadline. 3. Attacker manipulates block.timestamp to be slightly in the future, causing the transaction to revert with InvalidDeadline(). 4. This can be used for frontrunning or griefing attacks.

### [Medium] Fee-on-transfer tokens not handled in deposit()

**File:** `src/Redacted-Contract.sol` (line 158)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The deposit() function credits userBalances with the full \_amount before transferring tokens. If the token charges a fee on transfer, the contract receives less than credited, creating a balance mismatch.

**Exploit:** 1. Attacker creates a fee-on-transfer token (e.g., 10% fee). 2. Call deposit() with 100 tokens. 3. Contract credits userBalances with 100, but only receives 90 tokens. 4. Attacker can withdraw 100 tokens (if liquidity allows), draining 10 tokens from the pool.

**Recommendation:** Check actual balance change after transfer: uint256 balanceBefore = token.balanceOf(address(this)); token.safeTransferFrom(...); uint256 balanceAfter = token.balanceOf(address(this)); uint256 received = balanceAfter - balanceBefore; Credit userBalances with received amount.

**File:** `src/Redacted-Contract.sol` (line 253)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The \_onRandomNumberFulfilled() function is an internal override called by VRNGConsumer parent. However, there's no explicit verification that the callback is from the authorized VRNG system, relying entirely on parent contract implementation.

**Exploit:** 1. If VRNGConsumer parent has any vulnerability allowing unauthorized callbacks, attacker can trigger \_onRandomNumberFulfilled() with chosen randomness. 2. Attacker selects a favorable multiplier by controlling randomNumber input. 3. Claim inflated payouts from the liquidity pool.

**Recommendation:** Add explicit access control check in \_onRandomNumberFulfilled() to verify msg.sender is the authorized VRNG coordinator. Store and verify the VRNG system address.
