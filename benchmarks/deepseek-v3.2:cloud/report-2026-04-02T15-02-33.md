### [Critical] Fee-on-Transfer and Rebasing Tokens Cause Pool Insolvency

**File:** `src/Redacted-Contract.sol` (line 120)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The contract does not account for fee-on-transfer or rebasing token mechanics. When depositing or adding liquidity, the contract records the full `_amount` in user/pool balances, but the actual received amount may be less due to transfer fees. This discrepancy leads to an overstatement of internal accounting, making withdrawals and payouts impossible when the contract's actual token balance is lower than the recorded total liabilities.

**Exploit:** 1. Deploy a malicious ERC20 token with a 10% transfer fee. 2. Admin adds this token as a supported pool. 3. User deposits 100 tokens; contract receives only 90 but records 100 in userBalances. 4. User withdraws 100 tokens; contract attempts to transfer 100 but only holds 90, causing transaction to revert or draining other users' funds if extra liquidity exists. 5. Similarly, when the pool pays out a game win, the contract may not have enough tokens to cover the recorded payout amount.

**Recommendation:** Measure token balance before and after transfers to calculate the actual received amount. For deposits, use `balanceBefore = token.balanceOf(address(this))`; after transfer, `balanceAfter = token.balanceOf(address(this))`; then credit the difference to the user/pool.

### [High] Missing Reentrancy Guard on Random Number Callback

**File:** `src/Redacted-Contract.sol` (line 246)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The internal callback `_onRandomNumberFulfilled` is called by the external VRNG system and updates critical game state (multiplier, resultFulfilled) without a reentrancy guard. Although the parent functions have `nonReentrant`, the callback itself can be re-entered if the VRNG system is malicious or compromised, allowing state manipulation before the game is fully resolved.

**Exploit:** 1. Attacker creates a game and receives a requestId. 2. Malicious VRNG system calls `_onRandomNumberFulfilled` with a random number. 3. During the callback execution, before `resultFulfilled` is set to true, the VRNG system re-enters the contract via a function that reads the game state (e.g., a view function that returns the game's multiplier). 4. The game state is inconsistent (multiplier set but resultFulfilled still false), potentially allowing the attacker to claim an incorrect payout or disrupt game resolution.

**Recommendation:** Add a reentrancy guard modifier to `_onRandomNumberFulfilled` or ensure the VRNG system is trusted and cannot re-enter. Alternatively, use a mutex lock specific to the gameId to prevent reentrancy on a per-game basis.

### [Medium] Unbounded Loops Over Dynamic Arrays Risk Gas Limit Exhaustion

**File:** `src/Redacted-Contract.sol` (line 260)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** Multiple functions loop over dynamic arrays without size limits: `selectMultiplierFromRandom`, `addFeeRecipients`, `removeFeeRecipient`, and `payFees`. An admin (compromised or malicious) can push the array sizes to exceed block gas limits, causing DOS for critical operations like game resolution, fee distribution, and admin management.

**Exploit:** 1. Attacker compromises ADMIN_ROLE. 2. Adds thousands of fee recipients via `addFeeRecipients`. 3. When `createGame` is called, `payFees` iterates over the entire array, consuming all gas and reverting the transaction. 4. Similarly, adding many multiplier packages will cause `selectMultiplierFromRandom` to run out of gas during the VRNG callback, leaving games unresolved and funds locked.

**Recommendation:** Implement upper bounds on array sizes (e.g., max 50 fee recipients, max 100 multiplier packages). Add a `onlyRole(DEFAULT_ADMIN_ROLE)` for critical array expansions, or use pagination for loops in view functions.

### [Low] Floating Pragma Version Allows Inconsistent Compilation

**File:** `src/Redacted-Contract.sol` (line 2)
**Confidence:** 60%
**Flagged by:** auditor-1

**Description:** The pragma statement `^0.8.30` permits compilation with any Solidity version from 0.8.30 up to 0.9.0 (excluding). Different compiler versions may introduce subtle differences in bytecode generation, security checks, or optimization, leading to unpredictable behavior across deployments.

**Exploit:** A developer compiles the contract with Solidity 0.8.31, which contains a bug in the optimizer that incorrectly handles certain arithmetic operations. This could lead to incorrect calculations in fee distribution or multiplier selection, causing financial losses.

**Recommendation:** Lock the pragma to a specific version known to be secure, e.g., `pragma solidity 0.8.30;`. Use the same compiler version for all deployments and testing.
